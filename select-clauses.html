Documentation for the "selectClause" JSON key.

selectClause usage overview

JSON keyusagenotes
insertersinvalidInserters cannot have a selectClause
updatersinvalidUpdaters cannot have a selectClause
deletersinvalidDeleters cannot have a selectClause
findersoptional
questionsinvalidQuestions only ever return a single boolean object and as such cannot return a selectClause bean
countersinvalidCounters only ever return a single int object and as such cannot return a selectClause bean

By default finders return either a model or an ArrayList<Model> of models, which is directly bound to the database table on which the finders are generated. For example, a finder for the user table will return individual or a list of User model objects.

In some instances it may be necessary to choose which fields that are needed to be returned rather than the complete list. This is also necessary when you are selecting fields that come from corss tables, or you wish to use somw of the in built database functions that are present.

This is especially true when you have a blob or clob object that you do not wish to have returned in a list of models which will take up a lot of memory.

The generated bean has the following name <finder.name>Bean and is placed into the /bean directory.

Example 1: Subset of data from the user table.

This generates fields on the bean that are within the table definition.

{
	"name": "findNmUserDtmSignup",
	"selectClause": "select nm_user, dtm_signup from user",
	"selectFields": [
		{ "name": "nm_user", "type": "varchar" },
		{ "name": "dtm_signup", "type": "datetime" }
	],
	"unique": false
}

Example 2: Subset of data from with counts

This generates fields on the bean that are both within the table definition and utilise some in-built MySQL functions.

{
	"name": "findGroupNumAge",
	"selectClause": "select count(*) as num_count, num_age from user group by num_count",
	"selectFields": [
		{ "name": "num_count", "type": "int" },
		{ "name": "num_age", "type": "int" }
	],
	"unique": false
